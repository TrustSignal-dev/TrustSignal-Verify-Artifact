# TrustSignal in Plain English

> Legacy note: this setup guide is retained for the deprecated standalone action repository. Public installation guidance now belongs to `TrustSignal-dev/TrustSignal/github-actions/trustsignal-verify-artifact`.

This guide turns the TrustSignal "File Integrity Sidekick" deck into repo-accurate setup instructions for the published `TrustSignal Verify Artifact` action.

Use this when you want the shortest path from zero setup to a real verification run.

## Part 1: The GitHub Action

The GitHub Action runs inside your workflow YAML.

It does two jobs:

1. `local` mode
   Compare an artifact against a saved receipt without calling the API again.
2. `managed` mode
   Hash the artifact locally, call `https://api.trustsignal.dev`, and return live verification metadata.

### Before You Start: Get an API Key

If you only want offline receipt checks, you can skip this section and use `local` mode.

If you want live TrustSignal verification:

1. Request or obtain API access through the current TrustSignal onboarding path.
2. Generate an API key only through the authenticated TrustSignal surface when that path is deployed.
3. Open your GitHub repository.
4. Go to `Settings -> Secrets and variables -> Actions`.
5. Add a repository secret named `TRUSTSIGNAL_API_KEY`.
6. Paste the live key value.

Important:

- The GitHub secret alone is not enough for managed mode.
- The TrustSignal API backend must also allow that same key.
- The expected backend format is:

```text
API_KEYS=your-live-key
API_KEY_SCOPES=your-live-key=verify|read
```

If GitHub and the backend are out of sync, the action fails with `403 Forbidden: invalid API key`.

### Four Steps to Your First Verified Artifact

1. Create `.github/workflows/verify.yml`.
2. Build a real artifact in the workflow.
3. Add the TrustSignal action step.
4. Run the workflow and inspect the outputs.

Minimal managed example:

```yaml
name: Verify Artifact

on:
  workflow_dispatch:

jobs:
  verify:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Build release archive
        run: |
          mkdir -p build/package
          cp README.md build/package/README.md
          tar -czf build/trustsignal-demo-release.tgz -C build/package .

      - name: Verify with TrustSignal
        id: trustsignal
        uses: TrustSignal-dev/TrustSignal-Verify-Artifact@v0.2.0
        with:
          mode: managed
          path: build/trustsignal-demo-release.tgz
          api_base_url: https://api.trustsignal.dev
          api_key: ${{ secrets.TRUSTSIGNAL_API_KEY }}

      - name: Show outputs
        run: |
          echo "SHA256: ${{ steps.trustsignal.outputs.sha256 }}"
          echo "Status: ${{ steps.trustsignal.outputs.verification_status }}"
          echo "Receipt ID: ${{ steps.trustsignal.outputs.receipt_id }}"
          echo "Verification ID: ${{ steps.trustsignal.outputs.verification_id }}"
```

Minimal local example:

```yaml
- name: Create baseline receipt
  id: baseline
  uses: TrustSignal-dev/TrustSignal-Verify-Artifact@v0.2.0
  with:
    mode: local
    path: build/trustsignal-demo-release.tgz

- name: Verify against saved receipt
  uses: TrustSignal-dev/TrustSignal-Verify-Artifact@v0.2.0
  with:
    mode: local
    path: build/trustsignal-demo-release.tgz
    receipt: ${{ steps.baseline.outputs.receipt_path }}
```

### What the Outputs Mean

These are the outputs most people care about first:

| Output | Meaning |
| --- | --- |
| `verification_id` | Unique verification identifier |
| `verification_status` | Normalized result such as `verified` or `invalid` |
| `status` | Legacy alias for `verification_status` |
| `receipt_id` | Receipt identifier worth saving |
| `receipt_signature` | Cryptographic proof returned by the managed API |
| `sha256` | Digest of the artifact that was verified |
| `receipt_path` | Local receipt file path used or generated by the action |

### The Three Most Common Trip-Ups

1. Missing `api_base_url`
   Managed mode needs `https://api.trustsignal.dev` unless you are targeting another TrustSignal environment.
2. Wrong `path`
   Point the action at the actual file or directory you want hashed.
3. Artifact drift
   If the artifact changed after the receipt was created, local verification returns `invalid`.

For demos, a controlled drift example is expected. If you want to show the mismatch without failing the entire job, use:

```yaml
fail_on_mismatch: "false"
```

That keeps the step green while still surfacing the non-valid status in outputs.

## Part 2: The GitHub App

The GitHub App is different from the action.

The action runs inside a workflow you edit.

The app is a backend service you deploy once, connect to GitHub webhooks, and use to publish GitHub Check Runs automatically.

Use the app when you want org-wide monitoring without editing every workflow by hand.

### Five Steps to Deploy and Connect the App

1. Register the GitHub App with the required webhook URL, permissions, and events.
2. Deploy the TrustSignal app backend.
3. Set the required environment variables such as `GITHUB_APP_ID` and related private key settings.
4. Install the app on the repositories you want to monitor.
5. Push a commit and confirm the GitHub Check Run appears.

This repository focuses on the GitHub Action, but the app model matters because it answers the "when should I use which one?" question during onboarding and demos.

## Action vs. App

| Question | GitHub Action | GitHub App |
| --- | --- | --- |
| How it runs | Inside workflow YAML | Automatically via webhooks |
| Setup effort | Low | Medium |
| Best for | Per-workflow verification | Org-wide monitoring |
| Needs deployment | No | Yes |
| Shows GitHub Check Runs | No | Yes |

Recommended path:

1. Start with the action.
2. Prove the verification flow in one workflow.
3. Scale up to the app when you want centralized automation and Check Runs.
